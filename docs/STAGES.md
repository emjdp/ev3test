# 단계 로드맵 + 통과 기준(Done)

**원칙: 한 단계가 실기에서 "Done"을 만족하기 전에는 다음 단계로 가지 않는다.**
각 단계는 `stages/stageN_*.py` 하나의 독립 실행 스크립트로 구현한다. 그 단계가 쓰는
튜닝 값은 그 파일 맨 위 상수로 둔다(전역 거대 config 금지). 이전 단계에서 확정된
값/함수는 복사해 가져오거나 import 해서 재사용하되, **확정된 코드는 수정하지 않는다.**

각 단계 작업 후 [../PROGRESS.md](../PROGRESS.md)에 결과를 적고 커밋한다.

> **라이브 튜닝은 단계와 함께 자란다.** Stage 1부터 "파일 맨 위 상수"는 그 스테이지의
> *초기 params dict + PARAM_LIMITS* 가 되고, 노트북에서 실시간으로 바꾸며 튜닝한다.
> 단계마다 노출하는 params/telemetry 필드만 조금씩 늘린다(한 단계 6개 이하). 구조는
> [LIVE_TUNING.md](LIVE_TUNING.md). 단, **한 번에 변수 하나** 원칙은 그대로다.
>
> **Stage 1부터 모든 판단은 reason_code 로 기록한다**("110 코너 → 좌회전", "노드색 빨강
> → 도착"). 그래야 "왜 그렇게 움직였나"를 보고 *어떤 값을* 고칠지 안다. 판단 기록·재연·
> 단일동작 트리거(`robotctl do`)는 [DECISIONS.md](DECISIONS.md).

---

## Stage 0 — 연결/포트 확인

- **목표**: ev3dev 브릭에서 모터 3개·센서 4개가 설정한 포트로 인식되는지 확인.
- **구현**: 각 포트를 열어보고 성공/실패와 현재 센서값을 화면/콘솔에 출력. 모터는
  아주 짧게 저속으로 돌려 좌/우가 맞는지 눈으로 확인(선택).
- **Done**: 7개 장치가 모두 기대한 포트에서 에러 없이 열리고 값이 읽힌다.
  좌/우 모터 방향이 코드 기대와 일치한다.
- **실기 확정 메모(2026-06-30)**: Python 3.5.3, 좌/우 전진 방향 정상. `in1` 은 실행 로그에서
  미연결로 출력됐으나 사람이 실물 확인 후 Stage 0 통과로 확정했다(PROGRESS 기록 기준).

## Stage 1 — 기초 라인트레이싱 (+ 라이브 튜닝 infra 최초 도입)

- **목표**: 중앙 컬러센서 1개로 검은 선을 따라 직진·완만한 곡선을 안정적으로 추종.
  + 여기서 라이브 튜닝 최소 인프라(`lib/` + `tools/robotctl.py` get/set/stop)를 처음 만든다.
- **라이브 params(이 단계 것만)**: `kp`, `ki`, `kd`, `base_speed`, `turn_limit`,
  `target_reflect`(중앙센서 흑/백 중간 반사광). 좌우 모터 트림은 상수로 시작.
- **telemetry**: `reflect`, `error`, `turn`, `left_speed`, `right_speed`.
- **순서**: ① 흰 바닥/검은 선 반사광 raw 측정 → `target_reflect` 정함 → ② 저속 직진 쏠림을
  트림으로 잡음 → ③ 노트북에서 `kp` 올려가며 곡선 추종, 진동하면 내림(한 번에 하나).
- **Done**: 직선 + 곡선이 섞인 코스를 끝에서 끝까지 선을 벗어나지 않고 추종.
  `robotctl stop` 또는 키보드 인터럽트 등 BACK 이 아닌 정지 경로로 멈춘다.
- **하지 않는 것**: 분기 감지·회전·색 판정 없음, 대시보드/자동튜닝 없음. 선이 끊기면 그냥
  멈추거나 천천히 직진.

## Stage 2 — 원시 회전 (좌90 / 우90 / U턴)

- **목표**: 제자리(또는 한 바퀴) 회전으로 좌90°·우90°·180° U턴을 각각 일관되게 수행.
- **라이브 params**: `turn_speed`, 좌/우/U 각각 **엔코더 각도 + 보정계수**
  (`turn_90_factor`, `turn_180_factor`). 시간기반(ms)은 배터리/마찰에 가장 크게 흔들리니
  엔코더 각도 기반으로 두고 보정계수만 라이브로 만진다(LIVE_TUNING.md 기술결정 5).
- **보정 방식**: `robotctl do turn_left` 로 노트북에서 회전 1회 트리거 → 값 보고
  `turn_90_factor` 하나만 고침 → 다시 트리거. **재배포 없이** 도는 게 이 단계의 핵심
  (좌회전 하나에 1시간 쓰던 문제 해결). 회전마다 `TURN_*` reason 로그를 남긴다.
- **순서**: 좌90 한 동작만 반복 실행해 각도 맞춤 → 우90 → U턴. **셋을 따로** 맞춘다.
- **Done**: 바닥 표시 기준 각 회전을 3회 연속 ±몇 도 안에서 재현. (라인 재포착 방식은
  Stage 5에서 라인트레이싱과 합칠 때 도입; 여기서는 순수 회전 각도만 본다.)

## Stage 3 — 노드(분기) 감지 + 분기 회전

> **2026-07-02 확정**: 공식 구현체는 `stages/stage3v2_linetrace_branch.py`(bits+PD 라인추종 +
> `lib/turns.pivot` 탱크 회전). 이전에 검토했던 아날로그 centroid(`pos`/`total`) 설계와
> `stage3_node_detect.py`/`docs/specs/stage3_node_detect.md`(코드 미착수 상태였음)는 **폐기**
> 되었다 — 자세한 경위는 [PROGRESS.md](../PROGRESS.md) 2026-07-02 로그 참조.

- **목표**: 좌·중·우 3센서 raw 반사광 기반 PD 로 선을 추종하면서, 좌/우 3센서 흑/백
  bits(`LCR`, 예 `110`/`111`=좌 분기, `011`=우 분기)로 분기를 감지하면 **정지 → 교차점 위로
  전진(`branch_advance_mm`) → 제자리 탱크 90° 회전(`lib/turns.pivot`, Stage 2 확정 코드
  재사용) → 다음 선 재포착 후 계속 추종**까지 한다. (예전엔 "감지만 하고 멈춤"이 Stage 3,
  "감지 후 회전"은 Stage 5 몫이었다. v2 는 이 둘을 하나로 합쳤다 — 아래 Stage 5 메모 참조.)
- **라이브 params(6개)**: `kp`(조향 게인), `base_speed`, `turn_speed`(탱크 회전 속도),
  `turn_90_factor`(90° 보정계수, Stage 2 와 같은 방식), `branch_confirm_count`(분기 확정
  연속횟수, 오탐 방지), `branch_advance_mm`(확정 후 회전 전 전진거리 = 회전 시점). 센서
  threshold(좌43/중36/우42)·`kd`·`turn_limit`·`turn_180_factor` 등은 파일 상단 config 상수.
- **판단 기록**: `BRANCH_LEFT`/`BRANCH_RIGHT`(bits, branch_seen, advance_mm) — 분기 확정 시.
  회전 자체는 Stage 2 의 `TURN_LEFT`/`TURN_RIGHT`(target_deg, factor, enc_avg, error_deg) 를
  그대로 재사용해 기록한다. 라인추종 중엔 `LINE_FOLLOW`(throttle).
- **실패 #1 대응(분기/코너 오버슛)**: 너무 일찍 돌면 `branch_advance_mm` ↑, 직선에서 오회전하면
  `branch_confirm_count` ↑(오탐 방지가 먼저, 그다음 회전 시점 보정). `replay.py` 로 분기 확정
  타이밍(count/방향전환)을 로봇 없이 재연 검증.
- **Done**: 코스 위 좌/우 분기 각각에서 **제자리 90° 회전으로 다음 선에 올라타 계속 추종**하는
  것을 여러 번 재현하고, 주행 흔들림에 오회전하지 않는다.
- **안 하는 것(다음 단계로 미룸)**: 노드 종류(T자/십자/막다른 길) 구분, 미리 정한 방향
  **시퀀스**대로 도는 것(v2 는 "감지한 분기 쪽으로" 튼다), 색 판정 — Stage 5/4 로 남긴다.
- **메모**: Stage 1/2 에서 확정한 하드웨어/주행 기반(모터 부호 `left=base-turn`/`right=base+turn`,
  탱크 pivot 부호)은 그대로 재사용하며 수정하지 않는다(`lib/turns.pivot`, `lib/decide_turn.
  decide_turn`). 라인추종은 3센서 raw 좌/우 차 기반 PD(`stage3v2_linetrace_branch.py` 의
  `pd_step`)이며, 구 `decide_line3`(폐기된 아날로그/이전 bits-위치 방식) 는 더 이상 재사용
  대상이 아니다.

## Stage 4 — 색상코드 노드 판정

- **목표**: 막다른 길/노드에서 중앙센서를 컬러 모드로 읽어 노드 색을 판정
  (시작/체크포인트/도착 구분).
- **라이브 params**: 시작/체크포인트/도착 색 코드, 색 확정 샘플 수, 반사광↔컬러 전환
  settle/더미읽기, **색 읽는 위치(노드 확정 즉시 vs `node_advance` 후)**.
- **판단 기록**: `COLOR_READ` 에 `color`·`reflect`·`dist_since_node_mm` 를 남긴다.
- **실패 #2 대응(빈 바닥 색 측정)**: 노드를 지나 빈 바닥 색을 읽으면 오판한다. 색은
  **노드 확정 즉시(이동 전)** 읽고, 읽을 때 `reflect` 가 흰 바닥 수준이면 "바닥 읽음"
  경고를 reason 에 남겨 자동으로 잡는다. `do read_color` 로 위치별 재현.
- **Done**: 각 색 마커에서 의도한 색을 안정적으로 판정. 빈 바닥 오독 없음.
  전환 직후 오판(0/엉뚱한 색) 없음.

## Stage 5 — 통합: 라인트레이싱 + 노드에서 분기 회전

> **2026-07-02 메모**: Stage 3(v2)가 이미 "분기 감지 → 정지 → 전진 → 탱크 회전 → 재포착"을
> 한다(감지된 분기 쪽으로 자동으로 튼다). Stage 5 에 남는 몫은 ① **미리 정한/입력받은 방향
> 시퀀스**대로 도는 것(Stage 3 는 "보이는 분기 쪽" 고정) ② **노드 종류 구분**(T자/십자/막다른
> 길 — Stage 3 는 좌/우 분기만 본다)이다. 착수 시 Stage 3 의 `stage3v2_linetrace_branch.py`
> 함수(`black_bits`/`branch_side`/`pd_step`)를 그대로 재사용하고 새로 만들지 않는다.
>
> **2026-07-06 하위 단계 분할**: 통합 1방(`stage5_integration.py`)은 신규 변수가 많아 실기
> 디버깅이 안 됐다. **5-1(고정 지시 회전) → 5-2(111 노드 종류 구분) → 5-3(시퀀스 소비) →
> 5-4(LEAF 색 마커 + 전체 코스 = Stage 5 Done)** 로 쪼개, 하위 단계마다 신규 기능 하나씩
> 실기 Done 후 진행한다. 속도·회전 factor(`base_speed`/`turn_speed`/`turn_90_factor`)는
> 하위 단계 내내 라이브로 대시보드에 유지한다. 계획/Done 기준:
> [specs/stage5_substages.md](specs/stage5_substages.md).

- **목표**: 선 추종 → 노드 감지 → (정해진/입력된 방향) 회전 → 다음 선 올라타기 를 연결.
- **Done**: 미리 정한 회전 시퀀스(예: 좌,직,우,U)대로 코스를 노드마다 정확히 돌아 통과.

## Stage 6 — 노드 탐색/복귀

- **목표**: 지도 없이 모든 노드를 방문하고 도착까지 간 뒤 회전 기록을 역재생해 복귀.
- **Done**: 정해진 코스에서 전체 탐색 + 복귀 성공.

## Stage 7 — 물체 집기

- **목표**: 초음파로 전방 물체 감지 → 그리퍼로 집고 → 목표에 내려놓기.
- **Done**: 물체를 안정적으로 집어 운반·하차. 주행/센싱 방해 없음.
