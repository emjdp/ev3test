# Stage 3 v3 — 원속도 후보 위치(anchor) 보정 아이디어 메모

> 상태: DRAFT 아이디어 메모 (구현 미정, 실기 미검증)
> 선행: Stage 3 v2([stage3v2_linetrace_branch.md](stage3v2_linetrace_branch.md)) 실기 Done.
> 목적: Stage 3 v2 의 성공은 유지하되, `turn_90_factor=0.66` 으로
>       "라인추종 중 사전 회전 + 작은 pivot" 을 맞춘 편법을 나중에 제거할 수 있는 아이디어만
>       기록한다. **현 시점에 구현 의무 없음.**

## 0. 배경 / 이 문서의 위치

- Stage 3 v2 는 공식 Stage 3 구현체이며 실기 Done 상태다.
- 사용자가 확인한 v2 의 한계:
  - `110`/`011` 지점에서 분기 확정까지 라인트레이싱이 계속 돌면서 차체가 이미 조금 회전한다.
  - 그 뒤 제자리 pivot 을 그대로 돌리면 총 회전량이 커진다.
  - 그래서 `turn_90_factor` 를 0.66 으로 낮춰, "주행 중 생긴 사전 회전 + 작은 pivot" 으로
    실기 성공을 만들었다.
- 처음 논의한 "110/011 을 보면 즉시 정지 후 재확인" 방식은 정상 라인트레이싱 중 잠깐 보이는
  `110`/`011` 에도 자주 멈출 수 있어 설득력이 낮다. 이 문서는 그 안을 버리고, **원래 속도로
  주행하면서 판단하되 회전 기준 위치를 따로 보정하는 아이디어**만 남긴다.

## 1. 핵심 아이디어

원래 속도와 원래 PD 라인트레이싱을 유지한다. 대신 `110`/`011` 을 처음 본 순간의 위치를
`candidate_start` 로 저장해 둔다.

```
010 라인트레이싱 중 110/011 첫 감지
→ 아직 멈추거나 돌지 않음
→ candidate_side, candidate_start_enc 를 기록
→ 원래 속도/원래 PD 로 계속 진행
→ 짧은 거리 동안 bits 이력을 모아 진짜 분기인지 확정
→ 확정되면 정지
→ candidate_start 기준 원하는 pivot 위치를 계산
→ 너무 지나갔으면 필요한 만큼만 후진(backtrack)
→ 그 자리에서 Stage 2 pivot 으로 제자리 회전
```

즉 v2 의 문제를 `turn_90_factor` 로 숨기지 않고, **판단이 늦어진 거리**와 **회전해야 할 위치**를
분리해서 다룬다.

## 2. 후보 판단 개념

- `110` 이 처음 보이면 좌분기 후보, `011` 이 처음 보이면 우분기 후보로 잡는다.
- 후보가 생기면 encoder 기준으로 이후 짧은 거리의 bits 이력을 본다.
- 그 구간에서 같은 방향 패턴이 충분한 비율로 반복되면 진짜 분기로 확정한다.
- 반대 방향 패턴이 의미 있게 나오거나, 후보 패턴이 너무 짧게 끝나면 취소한다.
- 이 확인 중에도 주행은 v2 와 동일하게 원래 속도/원래 PD 를 유지한다.

## 3. 아이디어 파라미터 후보

아직 구현하지 않는다. 구현한다면 아래 중 최소 개수만 라이브로 연다.

| 이름 | 의미 | 출발점 |
|---|---|---|
| `branch_confirm_mm` | 후보 bits 이력을 볼 거리 | 20..35mm |
| `branch_confirm_ratio` | 같은 방향 패턴이 차지해야 하는 비율 | 0.6 |
| `branch_pivot_offset_mm` | 처음 감지한 지점에서 몇 mm 뒤를 회전 중심으로 볼지 | 5..15mm |
| `branch_backtrack_limit_mm` | 확정 후 너무 많이 후진하지 않도록 막는 상한 | 30mm |
| `turn_90_factor` | 순수 제자리 회전 보정값 | Stage 2 값 0.9 근처 |

`branch_confirm_count` 처럼 tick 수만 세면 속도/loop delay 에 흔들릴 수 있으므로, v3 를 실제로
한다면 거리(mm) 기반이 더 낫다.

## 4. 왜 이 방식이 더 현실적인가

- 순간적인 `110`/`011` 때문에 바로 멈추지 않는다.
- 기존 라인트레이싱 속도와 주행감을 유지한다.
- 확정이 늦어져도 처음 감지한 위치를 기억하므로, 회전 기준점을 나중에 보정할 수 있다.
- `turn_90_factor` 를 낮춰 사전 회전량까지 흡수하는 방식보다 원인과 보정값이 분리된다.

## 5. 구현하지 않는 이유 / 보류 조건

- 로직이 v2 보다 훨씬 복잡하다.
- 후진(backtrack)이 들어가면 구동 안정성, encoder 거리, 후진 중 선/분기 재감지 등 새 변수가 생긴다.
- Stage 3 v2 는 이미 실기 Done 이므로, 이 아이디어는 지금 당장 구현할 필요가 없다.
- Stage 5/6 에서 v2 방식이 다시 한계가 될 때만 이 문서를 꺼내 구현 여부를 판단한다.

## 6. 구현한다면 필요한 검증

- PC replay:
  - `110`/`011` 첫 감지 시점과 확정 시점 사이의 거리 분포 확인.
  - `branch_confirm_mm`/`ratio` 를 바꿨을 때 오탐/놓침이 어떻게 달라지는지 비교.
- 실기:
  - 확정 후 정지 거리와 필요한 backtrack 거리 측정.
  - backtrack 후 제자리 pivot 이 다음 선에 올라타는지 좌/우 각각 확인.
  - `turn_90_factor` 가 Stage 2 순수 회전값 근처로 돌아오는지 확인.
